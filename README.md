认证与安全模块
用户向控制器发起登录请求，请求参数携带用户名和密码。控制器接收请求后，调用业务层的login()方法，将登录参数转交业务层进行逻辑处理。
业务层调用PasswordEncoder组件，将用户输入的明文密码与数据库中存储的哈希加密密码进行比对校验。若用户密码校验失败，会向业务层返回fail结果；随后业务层将校验失败的结果回传给控制器，并由控制器向用户返回“用户名或者密码错误”的错误响应，登录流程终止。
若用户密码校验通过，组件将向业务层返回success结果。此时业务层调用Jwt服务的issueToken(id,username,role.name)方法，传入用户ID、用户名、角色名称信息，按照自定义规则生成JWT令牌。
业务层获取生成的token字符串后，将其传递给Controller完成数据封装，最终由控制器将JWT令牌返回给用户，完整登录流程执行完毕。

音乐检索与推荐模块
系统内所有音乐数据列表均支持分页加载展示，有效降低单次数据加载压力，避免页面数据冗杂过多导致加载缓慢，提升前端展示流畅度。系统内置关键词检索功能，检索条件可同时匹配歌曲标题、歌手名称进行综合模糊查询，检索精准度高，能够快速筛选出用户所需音乐资源。

播放器与播放队列模块
底部播放器和Pinia里的player负责当前播哪一首、列表、播放/暂停、音量等页面即时状态；播放模式（顺序/单曲/随机）也只存在前端，刷新后恢复默认。
已登录用户的播放队列存在数据库表user_play_queue（用户、歌曲、顺序），基于user_play_queue数据表实现持久化存储，通过/api/play-queue接口组完成数据管理。前端执行队列操作后自动同步至后端，页面加载与刷新时拉取数据库数据进行状态同步，确保播放队列不丢失。

歌单界面如图所示，用户可自主创建专属歌单，并可以对歌单内歌曲进行增删改查等日常维护操作。系统创建内置默认专属歌单“我喜欢的音乐”，登录后自动生成。收藏模块功能完善，支持歌曲收藏与取消收藏的开关式切换操作，操作响应流畅；同时实现收藏功能与自定义歌单的联动，用户收藏音乐时可直接添加至指定歌单，歌单内容变动也会同步更新收藏状态，实现数据统一与交互连续。

评星流程图如图5- 19所示，设计思路是当用户打开歌曲详情或底栏切到另一首歌时，前端请求GET/api/interaction/{音乐ID}/rating-stat。该接口允许未登录访问。服务端从汇总表读出各星人数、人数合计、平均分（由总分除以人数在查询中算出）；若请求里带有有效登录令牌，则再查当前用户是否已有评分，有则把“我的星级”一并返回。前端用返回数据渲染平均分、星级分布条。

所有上传文件的根文件夹，在项目根目录下的storage文件夹。其中music-dir: music规定歌曲音频文件存放路径为/storage/music/；cover-dir: covers规定歌曲封面图片存放路径为/storage/covers/；lyric-dir: lyrics规定歌词文件存放路径为/storage/lyrics/；avatar-dir: avatars规定用户头像图片存放路径为/storage/avatars/。

管理员上传表单提交音频与封面，服务端将文件存储于配置的根目录，路径写入music表。时长优先解析音频文件，回退表单字段。静态资源通过/files/暴露，使用绝对路径配置存储根目录避免路径错误。
